## Shared Types

These exact types must be used by both agents. Both copy verbatim. No interpretation.

### Backend — Pydantic schemas (`app/backend/app/schemas/feedback.py`)

```python
from pydantic import BaseModel, Field
from datetime import datetime
from enum import Enum
from typing import Optional


class Theme(str, Enum):
    UX = "UX"
    Performance = "Performance"
    Support = "Support"
    Pricing = "Pricing"
    Communication = "Communication"


class Status(str, Enum):
    Open = "Open"
    InProgress = "In Progress"
    Actioned = "Actioned"


class FeedbackCreate(BaseModel):
    client_name: str = Field(..., max_length=255)
    summary: str = Field(..., max_length=500)
    detail: Optional[str] = None
    theme: Theme
    status: Status = Status.Open


class FeedbackUpdate(BaseModel):
    client_name: Optional[str] = Field(None, max_length=255)
    summary: Optional[str] = Field(None, max_length=500)
    detail: Optional[str] = None
    theme: Optional[Theme] = None
    status: Optional[Status] = None


class FeedbackRead(BaseModel):
    id: int
    client_name: str
    summary: str
    detail: Optional[str] = None
    theme: Theme
    status: Status
    date_logged: datetime

    model_config = {"from_attributes": True}
```


**Backend service logic**:

No separate service layer needed for this feature — it is pure CRUD with no business logic beyond what the router pattern handles. Per architecture.md: "Services contain business logic." Since there is no scoring, validation beyond Pydantic, or cross-entity logic, putting CRUD directly in the router follows the existing briefs.py pattern exactly.

Router pseudocode:

1. **`list_feedback(db)`** → `GET /feedback/`
   - Query all Feedback rows ordered by `date_logged` DESC
   - Return list

2. **`get_feedback(feedback_id, db)`** → `GET /feedback/{id}`
   - Query Feedback by id
   - If not found → 404 "Feedback item not found"
   - Return single item

3. **`create_feedback(body: FeedbackCreate, db)`** → `POST /feedback/`
   - Instantiate Feedback from body.model_dump()
   - Add, commit, refresh
   - Return with status 201

4. **`update_feedback(feedback_id, body: FeedbackUpdate, db)`** → `PATCH /feedback/{id}`
   - Query Feedback by id
   - If not found → 404 "Feedback item not found"
   - Loop body.model_dump(exclude_unset=True), setattr each
   - Commit, refresh, return

5. **`delete_feedback(feedback_id, db)`** → `DELETE /feedback/{id}`
   - Query Feedback by id
   - If not found → 404 "Feedback item not found"
   - Delete, commit
   - Return 204 (no body)

**Frontend scope**: None in this slice.

**Contract endpoints covered**: All 5 — GET /, GET /{id}, POST /, PATCH /{id}, DELETE /{id}

**Frontend component tree** (this slice only):
```
NewFeedbackPage (server — renders form shell)
  ├── FormHeader (server — "Log Feedback" title, "Cancel" Link)
  └── FeedbackForm mode="create" (client — form state, validation, submit)

EditFeedbackPage (server — fetches getFeedback(id))
  ├── FormHeader (server — "Edit Feedback" title, "Cancel" Link)
  ├── FeedbackForm mode="edit" (client — pre-filled, validation, save + delete)
  └── NotFoundState (server — shown when fetch returns 404)
```

**FeedbackForm component logic (pseudocode)**:

1. **Props**: `mode: "create" | "edit"`, `initialData?: FeedbackItem`
2. **State**: `formData` (controlled inputs), `errors` (per-field), `loading` (submit in progress)
3. **Validation on blur**:
   - `client_name`: required, non-empty after trim
   - `summary`: required, non-empty after trim
   - `theme`: required, must be selected
   - (status and detail have no required validation)
4. **On submit (create mode)**:
   - Validate all required fields
   - If invalid → show errors, do not submit
   - Call `createFeedback(formData)`
   - On success → `router.push("/feedback")`
   - On error → show error message
5. **On submit (edit mode)**:
   - Same validation
   - Call `updateFeedback(id, formData)`
   - On success → `router.push("/feedback")`
6. **On delete (edit mode only)**:
   - `window.confirm("Are you sure you want to delete this feedback item?")`
   - If confirmed → call `deleteFeedback(id)`
   - On success → `router.push("/feedback")`

**Backend scope**: None in this slice.

**Contract endpoints covered by frontend**: GET /feedback/{id}, POST /feedback/, PATCH /feedback/{id}, DELETE /feedback/{id}

## Decisions Made

1. **No separate service layer for feedback CRUD**
   Why: The feature is pure CRUD with no business logic (no scoring, no cross-entity operations, no computed fields). The existing codebase pattern (briefs.py) puts simple CRUD directly in the router. Adding a service layer would be empty pass-through functions.
   Alternative rejected: Creating `app/backend/app/services/feedback.py` — rejected because it adds indirection with no benefit for simple CRUD. If business logic is added later (e.g. auto-categorization), a service can be extracted then.

2. **Client-side filtering only (no server query params)**
   Why: The ui-spec explicitly states "Filtering is client-side and immediate (no submit button)." The contract defines GET /feedback/ with no query parameters. For MVP data volumes this is correct.
   Alternative rejected: Server-side filtering with query params — rejected because it contradicts the approved contract and ui-spec. Can be added later if data volume requires it.

3. **FeedbackListClient as a client component wrapper**
   Why: The filter state (search text, theme, status) requires useState hooks. Next.js server components cannot use hooks. The page itself is a server component that fetches data, then passes it to a client component that manages interactive filtering.
   Alternative rejected: Making the entire page a client component — rejected because it loses server-side data fetching benefits and contradicts Next.js app router patterns.

4. **Single FeedbackForm component for both create and edit modes**
   Why: The ui-spec shows identical fields for both modes. A single component with a `mode` prop avoids duplication. The edit page pre-fills `initialData`, the create page does not.
   Alternative rejected: Separate CreateFeedbackForm and EditFeedbackForm — rejected because the form fields are identical, leading to code duplication.

5. **Theme and Status as string enums (not integer/DB enum type)**
   Why: The contract specifies human-readable string values ("UX", "In Progress"). Using SAEnum with str-based Python enums keeps the DB values readable and matches the API contract directly. No mapping layer needed.
   Alternative rejected: Integer enums with a mapping table — rejected because it adds complexity for 5 fixed values that the PM confirmed are fixed for MVP.

6. **`date_logged` instead of `created_at` for the timestamp column name**
   Why: The contract and ui-spec both use `date_logged` as the field name. While the data-patterns.md convention is `created_at`, the contract is the source of truth for API field names. The SQLAlchemy column is named `date_logged` so `from_attributes` maps directly without aliasing.
   Alternative rejected: Name the column `created_at` and alias in the schema — rejected because it adds confusion between the DB column name and the API field name for no benefit.

7. **`updated_at` column included despite not being in the contract**
   Why: The data-patterns.md convention says "Every model has: id, created_at, updated_at." The `updated_at` column is standard for auditability. It is not exposed in the API response (FeedbackRead does not include it) so it does not affect the contract.
   Alternative rejected: Omitting `updated_at` — rejected because it breaks the standard model pattern and removes useful audit data.

8. **Browser `confirm()` for delete (not a styled modal)**
   Why: The ui-spec explicitly says "triggers browser confirm dialog." The PM designed it this way. Using a styled modal would deviate from the approved design.
   Alternative rejected: Custom confirmation modal — rejected because it contradicts the approved ui-spec.

9. **Frontend component directory: `components/feedback/` not `components/forms/` + `components/board/`**
   Why: The existing architecture shows `components/board/` and `components/forms/` for the story assignment domain. For the feedback feature, a single `components/feedback/` directory is cleaner since all components belong to one feature. This avoids spreading related components across unrelated directories.
   Alternative rejected: Using existing `board/` and `forms/` directories — rejected because those are semantically for the story assignment domain, not feedback.

10. **`FeedbackForm` handles delete button internally (edit mode)**
    Why: The ui-spec shows the Delete button as part of the edit form page, below the Save button. It's simpler to include it in the form component (shown only in edit mode) than to create a separate DeleteButton component that needs the same ID and navigation logic.
    Alternative rejected: Separate DeleteButton component — rejected because it adds an extra component for a single button that is contextually tied to the edit form.

## Testability Notes

- **All 5 endpoints are straightforward CRUD** — no external dependencies, no async operations, no background tasks. The SQLite test DB pattern from conftest.py works directly.
- **`date_logged` uses `server_default=func.now()`** — tests cannot control this value. Tests that check ordering must create items with a small sleep or use database-level timestamp injection. The tester agent should be aware of this.
- **FeedbackUpdate with `exclude_unset=True`** — the PATCH endpoint distinguishes between "field not sent" and "field sent as null." The `detail` field can be explicitly set to `null` to clear it, vs. omitting it to leave it unchanged. Tester should cover both cases.
- **Enum validation is handled by Pydantic** — sending an invalid theme or status value returns 422 automatically. No custom validation code to test, but tester should verify the error shape.
- **Frontend filtering is pure client-side logic** — no API interaction. Can be tested with component tests if needed, but the tester agent focuses on API integration tests per the task list.

## Open Questions

1. **Should `detail` field accept empty string or only null?**
   Default if not answered: Accept both. Empty string `""` is stored as-is. `null` means no detail. The FeedbackUpdate schema allows setting detail to `null` to clear it. The frontend sends `null` for empty textarea (not empty string).

2. **Maximum length for `detail` field?**
   Default if not answered: No maximum enforced. The `detail` field is `Text` type in the database (unlimited) and has no `max_length` in the Pydantic schema. The contract does not specify a limit. If PM wants a limit, it can be added to the schema later.

3. **Should the frontend show loading states for API calls?**
   Default if not answered: Yes. The FeedbackForm shows a disabled submit button with "Saving..." text during submission. The FeedbackListPage shows the table immediately (server-side fetch). No skeleton loading states — the server component handles this.

4. **CORS allowed origins for production?**
   Default if not answered: Development only for now — `http://localhost:3000`. The docker-compose sets `NEXT_PUBLIC_API_URL=http://localhost:8000`. Production CORS config is out of scope for MVP.
