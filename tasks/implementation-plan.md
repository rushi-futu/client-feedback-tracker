## Data Model

### New tables

| Table | Column | Type | Constraints | Notes |
|-------|--------|------|-------------|-------|
| feedback_items | id | Uuid | PK, `default=uuid.uuid4` | Python-generated UUID4; uses SQLAlchemy 2.0 `Uuid` type for cross-DB compat |
| feedback_items | client_name | VARCHAR(255) | NOT NULL | Free text — validated non-empty by schema |
| feedback_items | summary | VARCHAR(500) | NOT NULL | Short description — validated non-empty by schema |
| feedback_items | detail | TEXT | nullable | Optional; NULL means not provided |
| feedback_items | theme | VARCHAR(20) | NOT NULL | SAEnum with `native_enum=False`; values: ux / performance / support / pricing / communication |
| feedback_items | status | VARCHAR(20) | NOT NULL, default `'open'` | SAEnum with `native_enum=False`; values: open / in_progress / actioned |
| feedback_items | created_at | TIMESTAMPTZ | NOT NULL, `server_default=func.now()` | Set by DB on INSERT; never written by application |
| feedback_items | updated_at | TIMESTAMPTZ | NOT NULL, `server_default=func.now()` | Application must explicitly set to `datetime.now(UTC)` on PATCH — see Decision D5 |

### Backend — Pydantic schemas (`backend/app/schemas/feedback_item.py`)

```python
from pydantic import BaseModel, field_validator
from datetime import datetime
from enum import Enum
from uuid import UUID


class ThemeEnum(str, Enum):
    ux = "ux"
    performance = "performance"
    support = "support"
    pricing = "pricing"
    communication = "communication"


class StatusEnum(str, Enum):
    open = "open"
    in_progress = "in_progress"
    actioned = "actioned"


class FeedbackItemCreate(BaseModel):
    client_name: str
    summary: str
    detail: str | None = None
    theme: ThemeEnum
    status: StatusEnum = StatusEnum.open

    @field_validator("client_name", "summary")
    @classmethod
    def must_not_be_empty(cls, v: str) -> str:
        if not v.strip():
            raise ValueError("must not be empty or whitespace")
        return v


class FeedbackItemUpdate(BaseModel):
    client_name: str | None = None
    summary: str | None = None
    detail: str | None = None
    theme: ThemeEnum | None = None
    status: StatusEnum | None = None

    @field_validator("client_name", "summary")
    @classmethod
    def must_not_be_empty_if_provided(cls, v: str | None) -> str | None:
        if v is not None and not v.strip():
            raise ValueError("must not be empty or whitespace if provided")
        return v


class FeedbackItemRead(BaseModel):
    id: UUID
    client_name: str
    summary: str
    detail: str | None
    theme: ThemeEnum
    status: StatusEnum
    created_at: datetime
    updated_at: datetime

    model_config = {"from_attributes": True}
```

#### Backend model spec — `backend/app/models/feedback_item.py`

```
imports: Column, String, Text, DateTime, Enum as SAEnum, Uuid (sqlalchemy.types)
         func (sqlalchemy.sql), uuid (stdlib), Base (app.database)
         ThemeEnum, StatusEnum (app.schemas.feedback_item)

class FeedbackItem(Base):
    __tablename__ = "feedback_items"

    id         = Column(Uuid, primary_key=True, default=uuid.uuid4, index=True)
    client_name = Column(String(255), nullable=False)
    summary    = Column(String(500), nullable=False)
    detail     = Column(Text, nullable=True)
    theme      = Column(SAEnum(ThemeEnum, native_enum=False, length=20), nullable=False)
    status     = Column(SAEnum(StatusEnum, native_enum=False, length=20),
                        nullable=False, default=StatusEnum.open)
    created_at = Column(DateTime(timezone=True), server_default=func.now(), nullable=False)
    updated_at = Column(DateTime(timezone=True), server_default=func.now(), nullable=False)
    # NOTE: updated_at has NO onupdate= — the router sets it explicitly on PATCH (see Decision D5)
```

#### Backend router spec — `backend/app/routers/feedback.py`

Router prefix: `/feedback`, tag: `feedback`

**`GET /feedback/`** → `list_feedback(db)`
```
query feedback_items ordered by created_at DESC
return list of all items
```

**`POST /feedback/`** → `create_feedback(body: FeedbackItemCreate, db)` — status 201
```
item = FeedbackItem(**body.model_dump())
db.add(item)
db.commit()
db.refresh(item)
return item
```

**`GET /feedback/{item_id}`** → `get_feedback(item_id: UUID, db)` — status 200 or 404
```
item = db.query(FeedbackItem).filter(FeedbackItem.id == item_id).first()
if not item: raise HTTPException(404, "Feedback item not found")
return item
```

**`PATCH /feedback/{item_id}`** → `update_feedback(item_id: UUID, body: FeedbackItemUpdate, db)` — status 200 or 404
```
item = db.query(FeedbackItem).filter(FeedbackItem.id == item_id).first()
if not item: raise HTTPException(404, "Feedback item not found")
updates = body.model_dump(exclude_unset=True)
for key, value in updates.items():
    setattr(item, key, value)
item.updated_at = datetime.now(timezone.utc)   # explicit — see Decision D5
db.commit()
db.refresh(item)
return item
```

**`DELETE /feedback/{item_id}`** → `delete_feedback(item_id: UUID, db)` — status 204
```
item = db.query(FeedbackItem).filter(FeedbackItem.id == item_id).first()
if not item: raise HTTPException(404, "Feedback item not found")
db.delete(item)
db.commit()
return (no body — 204 No Content)
```

## Decisions Made

1. **No services layer — router calls DB directly**
   Why: The feature is pure CRUD with no business logic to extract. The architecture
   pattern says "services contain business logic" — there is none here. Adding a services
   layer would be indirection with no benefit and would split logic the tester needs to
   see in one place.
   Alternative rejected: Creating `backend/app/services/feedback.py` with pass-through
   functions. Rejected because it adds files and complexity without any isolation benefit
   when the functions would be trivially thin wrappers around ORM calls.

2. **`native_enum=False` on SAEnum columns (theme, status)**
   Why: Stores enum values as VARCHAR strings in both PostgreSQL and SQLite. This ensures
   the test database (SQLite) works identically to production (PostgreSQL) without needing
   PostgreSQL-specific enum CREATE TYPE DDL. Adds minor read overhead, immaterial at this scale.
   Alternative rejected: `native_enum=True` (PostgreSQL ENUM type). Rejected because it
   requires `CREATE TYPE` DDL that SQLite doesn't support, breaking the test conftest pattern.

3. **`updated_at` set explicitly in PATCH router, not via `onupdate=func.now()`**
   Why: SQLAlchemy's `onupdate` fires a Python-side function only when the ORM detects a
   dirty state on that column. With `setattr` + `db.commit()`, the `onupdate` trigger may
   not fire reliably in SQLite (SQLite has no server-side `now()` function; `func.now()`
   falls back to a Python datetime that the ORM may not register as a model event). Explicit
   assignment of `datetime.now(timezone.utc)` to `item.updated_at` before commit is
   guaranteed to work in both PostgreSQL and SQLite test environments.
   Alternative rejected: `onupdate=func.now()` alone. Rejected because of the SQLite
   incompatibility noted above, which would cause T-04 (updated_at must change on PATCH)
   to flap between environments.

4. **`FeedbackListClient` client wrapper component for filter state**
   Why: `FeedbackListPage` (server component) cannot hold React state, but the FilterBar
   requires `useState` to control its inputs. The Next.js 15 pattern is to pass server-fetched
   data as a prop to a client wrapper that manages interactive state. This avoids re-fetching
   on every filter change (all filtering is client-side per the ui-spec) and keeps the data
   fetch in the server component.
   Alternative rejected: Making `FeedbackListPage` itself a client component. Rejected because
   it would convert the data fetch to a client-side `useEffect`, adding a loading flash and
   network waterfall on page load, violating Next.js 15 server-first conventions.

5. **UUID as Python-level default, not a DB server default**
   Why: `default=uuid.uuid4` in the SQLAlchemy column means Python generates the UUID before
   the INSERT, so the value is known immediately after `db.add()` without needing `db.refresh()`.
   Using `gen_random_uuid()` as a server default would require a `db.refresh()` call and a
   PostgreSQL-specific function not available in SQLite.
   Alternative rejected: `server_default=text("gen_random_uuid()")`. Rejected because of
   SQLite incompatibility in tests and the extra round-trip to retrieve the generated ID.

6. **FeedbackForm sends only changed fields on PATCH (diff against `initialData`)**
   Why: The contract specifies PATCH semantics — "only changed fields need to be sent".
   Sending all fields on every update would overwrite any concurrent changes and unnecessarily
   move data. The `exclude_unset=True` pattern on the backend reinforces this — unset fields
   are ignored.
   Alternative rejected: Always send all fields (PUT-style PATCH). Rejected as inconsistent
   with the contract's PATCH semantics and `FeedbackItemUpdate` where all fields are Optional.

## Testability Notes

1. **`updated_at` on PATCH** — Decision D3 ensures this works in SQLite tests. The tester
   (T-04) must verify that a PATCH response has a different `updated_at` than the pre-PATCH
   GET. This is a required assertion, not advisory.

2. **UUID in SQLite** — SQLAlchemy's `Uuid` type stores UUIDs as strings in SQLite. The
   tester should assert that `id` in responses is a valid UUID string (36 chars, hyphenated).
   FastAPI serialises `UUID` objects as strings automatically.

3. **Empty string validation (T-02 adversarial)** — The `@field_validator` on
   `FeedbackItemCreate.client_name` and `summary` fires on `" "` (whitespace-only) as well
   as `""`. Tester must include both cases: `""` → 422, `"   "` → 422.

4. **Empty PATCH body (T-04 adversarial)** — `{}` body with `exclude_unset=True` applies
   zero changes but still sets `updated_at`. The response must be 200 with the existing
   data unchanged (except potentially `updated_at` advancing by milliseconds).

5. **Delete idempotency (T-05)** — Second DELETE must return 404, not 204. The router
   raises 404 immediately if the item is not found, so this is naturally correct but must
   be verified explicitly.

6. **`window.confirm` in FeedbackForm delete** — Cannot be tested in the backend suite.
   Frontend E2E tests (out of scope) would need to mock `window.confirm`. The tester agent
   should note this as a manual verification item.

7. **FilterBar filtering is fully client-side** — No backend test needed for filtering.
   Frontend unit tests (if written) should test `FeedbackListClient` filter logic in
   isolation, passing a mock items array and asserting the filtered output.

## Open Questions ⚠️

1. **Should empty `detail` string (`""`) be stored as NULL or as an empty string?**
   The `detail` field is `TEXT nullable`. If the user clears the Detail textarea and submits,
   `FeedbackItemCreate.detail` will be `""` (empty string) or omitted.
   Default if not answered: Treat `""` as `None` — the form sends `detail: null` (not `""`)
   when the textarea is empty; the FeedbackForm component should convert empty string to
   `null` before sending the API payload.

2. **Maximum number of feedback items before performance degrades?**
   The spec says no pagination. `GET /feedback/` returns all items. At what volume does
   the "load all on page mount" approach become a problem?
   Default if not answered: No limit enforced. Ship without pagination. Add if the count
   exceeds 500 items in production monitoring.

3. **Should the API URL (`NEXT_PUBLIC_API_URL`) be configured at build time or runtime?**
   Default if not answered: Build-time via `NEXT_PUBLIC_API_URL` env variable (standard
   Next.js public env var). Defaults to `http://localhost:8000` if not set. This is the
   pattern shown in `frontend-patterns.md`.
