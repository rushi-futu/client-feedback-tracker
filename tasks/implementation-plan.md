**Backend service logic (pseudocode — no real code)**:

All business logic lives directly in the router functions (CRUD is thin; no separate service layer needed).

```
function list_feedback(db):
    query feedback_items
    ORDER BY created_at DESC
    return list

function create_feedback(body: FeedbackItemCreate, db):
    construct FeedbackItem from body fields
    db.add(item)
    db.commit()
    db.refresh(item)
    return item

function get_feedback(id: UUID, db):
    query feedback_items WHERE id = id
    if not found: raise 404 "Feedback item not found"
    return item

function update_feedback(id: UUID, body: FeedbackItemUpdate, db):
    query feedback_items WHERE id = id
    if not found: raise 404 "Feedback item not found"
    updates = body.model_dump(exclude_unset=True)
    for each (key, value) in updates:
        if value is not None:
            setattr(item, key, value)
        elif key == "detail":
            # detail: null explicitly clears the field
            setattr(item, key, None)
        # else: non-detail field with null value — skip, do not update
    db.commit()
    db.refresh(item)
    return item

function delete_feedback(id: UUID, db):
    query feedback_items WHERE id = id
    if not found: raise 404 "Feedback item not found"
    db.delete(item)
    db.commit()
    return (no content)
```

**UUID type**: Use `sqlalchemy.types.Uuid(as_uuid=True)` from SQLAlchemy 2.0 — this is dialect-agnostic (maps to PostgreSQL native UUID; maps to CHAR(32) in SQLite for tests). Default on the column: `default=uuid.uuid4`.

**Router prefix**: `/feedback`, tags `["feedback"]`. Registered in `main.py` via `app.include_router(feedback.router)`.

**CORS**: Allow origin `http://localhost:3000` (development). Allow all methods and headers.

## Decisions Made

1. **No separate service layer — router handles CRUD directly**
   Why: The feature is pure CRUD with no complex business logic. The api-patterns.md reference implementation (briefs router) shows DB queries directly in the router. Introducing a service layer for five simple operations would add files with no benefit.
   Alternative rejected: `backend/app/services/feedback.py` — adds indirection with no payoff at this complexity level.

2. **UUIDs use `sqlalchemy.types.Uuid(as_uuid=True)` (SQLAlchemy 2.0 native)**
   Why: Dialect-agnostic — works in both PostgreSQL (production) and SQLite (test DB per conftest.py pattern). Avoids importing `sqlalchemy.dialects.postgresql.UUID` which would fail in SQLite test environment.
   Alternative rejected: `String(36)` — works everywhere but gives up DB-level UUID type enforcement in PostgreSQL, and complicates Python-side handling (strings vs UUID objects).

3. **Client-side filtering via `FeedbackListClient` wrapper component**
   Why: ui-spec.md explicitly states "Filters on /feedback are client-side — no round trip needed if all items loaded." The server component (`FeedbackListPage`) fetches all items once; the client wrapper (`FeedbackListClient`) owns the filter state and derived filtered list. This keeps the page server-rendered for initial load while making filter interactions instant.
   Alternative rejected: Making FeedbackListPage a client component that fetches on mount — loses server-rendering benefits and adds `useEffect` complexity unnecessarily.

4. **FeedbackForm is a single component for both create and edit modes**
   Why: Both modes share identical fields; only the submit label, submit action, and presence of the delete button differ. A single component with `mode: "create" | "edit"` and optional `defaultValues` is simpler and easier to maintain than two separate form components.
   Alternative rejected: `CreateFeedbackForm` + `EditFeedbackForm` — code duplication for identical field set.

5. **Delete button lives inside FeedbackForm (edit mode only)**
   Why: FeedbackForm is already a client component with `useRouter`. Placing the delete button inside FeedbackForm (conditionally rendered on `mode === "edit"`) avoids needing a second client component on FeedbackDetailPage. FeedbackDetailPage stays a clean server component.
   Alternative rejected: Separate `DeleteButton` client component on FeedbackDetailPage — adds a file and complicates the component tree for no gain.

6. **PATCH null-handling: skip null for non-detail fields; honour null for detail**
   Why: The contract specifies "only provided (non-null) fields are updated" with the exception "To clear detail, send detail: null explicitly." This asymmetry is encoded in the update_feedback pseudocode. `exclude_unset=True` ensures fields absent from the request body are not in the updates dict at all.
   Alternative rejected: `exclude_none=True` across all fields — would silently prevent clearing `detail` via `null`.

7. **No pagination — all items fetched on page load**
   Why: ui-spec.md explicitly states "No pagination — all items loaded on page mount." This is a tool for a small team; scale is not a current concern.
   Alternative rejected: Cursor or offset pagination — out of scope per PM decision.

8. **Greenfield project — all bootstrap files created in Slice 1 (backend) and Slice 2 (frontend)**
   Why: No existing backend or frontend directories exist. Both agents must create the full project scaffold including pyproject.toml, next.config.ts, etc.
   Alternative rejected: Assuming scaffold files exist — they do not.

## Testability Notes

- **Router is thin and CRUD-only** — all five endpoints are straightforward to test via `TestClient` with the SQLite in-memory test DB using the `conftest.py` pattern from data-patterns.md. No mocking needed.

- **`updated_at` with SQLite** — `onupdate=func.now()` in SQLAlchemy generates `CURRENT_TIMESTAMP` in the UPDATE SQL statement. SQLite supports `CURRENT_TIMESTAMP` but its resolution is seconds, not microseconds. T-04 (verify `updated_at` changes) is valid in SQLite but must account for possible sub-second equality if the test runs within the same second. Tester should add a brief `time.sleep(1)` or use PostgreSQL for that specific assertion. Flag for tester agent.

- **UUID primary key in SQLite** — `sqlalchemy.types.Uuid(as_uuid=True)` stores as CHAR(32) in SQLite. Querying by UUID object should work transparently via SQLAlchemy. Tester should pass actual `uuid.UUID` objects or string UUIDs consistently.

- **PATCH semantics (T-08)** — to verify that unset fields are not modified, the test should: create an item, PATCH only `status`, then GET the item and assert `client_name` and `summary` are unchanged.

- **Client-side filtering** — there is no server-side filter logic to test. The GET /feedback/ endpoint always returns all items sorted by `created_at` DESC. Filter permutations are frontend concerns only.

- **Frontend tests** — not in scope for the backend build agent. Tester agent handles these separately per T-series tasks.

## Open Questions ⚠️

1. **`detail` clearing via PATCH null: does the frontend ever need to clear `detail`?**
   The contract supports it. The FeedbackForm sends `detail: ""` (empty string) or `detail: null` depending on how the textarea is cleared. Decision needed: does an empty textarea in edit mode send `detail: null` (clear) or `detail: ""` (set to empty string)?
   **Default if not answered**: Frontend sends `detail: null` when the detail textarea is empty in edit mode (clearing the field). Empty string is not a valid "clear" signal.

2. **Status display labels in FeedbackTable: raw enum values or human-friendly labels?**
   The enum values are `open`, `in_progress`, `actioned`. The ui-spec does not specify display formatting.
   **Default if not answered**: Display with basic capitalisation and underscore-to-space conversion:
   - `open` → "Open"
   - `in_progress` → "In Progress"
   - `actioned` → "Actioned"

3. **Theme display labels in FeedbackTable: are they displayed exactly as the enum values?**
   The enum values (UX, Performance, Support, Pricing, Communication) are already human-readable.
   **Default if not answered**: Display exactly as enum values — no transformation needed.

4. **Environment variable for API base URL in production frontend deployment?**
   The frontend reads `NEXT_PUBLIC_API_URL`. No production URL is defined anywhere in the harness config.
   **Default if not answered**: Default to `http://localhost:8000` for development. Deployment env config is out of scope for this plan — a `.env.local.example` file will be created for guidance.
