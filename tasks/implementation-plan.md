## Shared Types

These exact types must be used by both agents. Both copy verbatim. No interpretation.

### Backend — Pydantic schemas (backend/app/schemas/feedback.py)

```python
from pydantic import BaseModel
from datetime import datetime
from enum import Enum
from uuid import UUID


class Theme(str, Enum):
    UX = "UX"
    Performance = "Performance"
    Support = "Support"
    Pricing = "Pricing"
    Communication = "Communication"


class FeedbackStatus(str, Enum):
    open = "open"
    in_progress = "in_progress"
    actioned = "actioned"


class FeedbackCreate(BaseModel):
    client_name: str
    summary: str
    detail: str | None = None
    theme: Theme
    status: FeedbackStatus


class FeedbackUpdate(BaseModel):
    client_name: str | None = None
    summary: str | None = None
    detail: str | None = None
    theme: Theme | None = None
    status: FeedbackStatus | None = None


class FeedbackRead(BaseModel):
    id: UUID
    client_name: str
    summary: str
    detail: str | None
    theme: Theme
    status: FeedbackStatus
    created_at: datetime
    updated_at: datetime

    model_config = {"from_attributes": True}
```

**Backend service logic**:

No dedicated service layer for Slice 1. This is pure CRUD with no business logic beyond validation. All logic lives in the router, following the existing pattern from api-patterns.md.

Router functions (pseudocode):

- `list_feedback(db)` → query all FeedbackItem, return as list. No filtering (client-side per ui-spec).
- `get_feedback(feedback_id: UUID, db)` → query by id, raise 404 if not found, return single item.
- `create_feedback(body: FeedbackCreate, db)` → instantiate FeedbackItem from body.model_dump(), add to session, commit, refresh, return.
- `update_feedback(feedback_id: UUID, body: FeedbackUpdate, db)` → query by id, raise 404 if not found, iterate body.model_dump(exclude_unset=True), setattr each, commit, refresh, return.
- `delete_feedback(feedback_id: UUID, db)` → query by id, raise 404 if not found, delete, commit, return 204.

## Decisions Made

1. **UUID primary key using `uuid.uuid4` with PostgreSQL `gen_random_uuid()`**
   Why: Contract specifies `id: uuid`. PostgreSQL has native UUID support. Using `gen_random_uuid()` as server default so IDs are generated at the DB level.
   Alternative rejected: Application-side UUID generation — adds unnecessary Python dependency, DB-level is simpler and consistent.

2. **No dedicated service layer for this feature**
   Why: Pure CRUD with no business logic. Adding a service layer would be unnecessary indirection. The router handles DB operations directly, following the existing api-patterns.md pattern.
   Alternative rejected: Service layer wrapping CRUD — would be an empty pass-through. Can be introduced later if business rules emerge.

3. **Server component page with client component wrapper for FeedbackListPage**
   Why: The page fetches data (server), but filtering is interactive (client). A client wrapper component (FeedbackListClient) receives all items as props and manages filter state locally. This follows Next.js 15 best practice — server components for data, client components for interactivity.
   Alternative rejected: Making the entire page a client component — loses server-side rendering benefits and doesn't follow the Next.js app router patterns.

4. **Client-side filtering with all items loaded at once**
   Why: Explicitly stated in ui-spec: "Filters on `/feedback` are client-side — no round trip needed if all items loaded" and "No pagination — all items loaded on page mount". Simple filter logic in the client wrapper.
   Alternative rejected: Server-side filtering with query params — contradicts the ui-spec.

5. **Single FeedbackForm component for both create and edit modes**
   Why: Both modes have identical fields per ui-spec. A `mode` prop controls submit button text, pre-population, and which API call to make. Reduces duplication.
   Alternative rejected: Separate CreateFeedbackForm and EditFeedbackForm — violates DRY, both forms are identical per spec.

6. **Delete button on FeedbackDetailPage, not in FeedbackForm**
   Why: Delete is a page-level action (ui-spec: "Delete button — bottom of form, secondary/destructive style") with its own confirm flow. Keeping it outside the form prevents accidental submission confusion.
   Alternative rejected: Delete button inside FeedbackForm with a mode check — mixes form submission logic with destructive actions.

7. **Theme enum values are PascalCase ("UX", "Performance") not lowercase**
   Why: Contract specifies `values: [UX, Performance, Support, Pricing, Communication]` with those exact casings. Both backend Enum and frontend type union use these exact values.
   Alternative rejected: Lowercase theme values — would contradict the contract.

8. **Frontend component directory: `components/feedback/` not `components/forms/` + `components/board/`**
   Why: The existing architecture shows domain-grouped components (`board/`, `forms/`). For this feature, all components are feedback-specific. Using `components/feedback/` groups them by domain, which is cleaner for a single-domain feature. FilterBar, FeedbackTable, and FeedbackForm all live here.
   Alternative rejected: Splitting across `forms/` and `board/` — unnecessary for a single domain with only 3 components.

## Testability Notes

- **All router functions are straightforward CRUD** — no complex branching, no external calls. Easy to test with TestClient + SQLite.
- **UUID primary keys** — tests must capture the `id` from POST response to use in subsequent GET/PATCH/DELETE calls. Cannot use hardcoded IDs.
- **FeedbackUpdate partial updates** — the `exclude_unset=True` pattern means tests should verify that sending `{"status": "actioned"}` only changes status, not other fields. This is a common source of bugs.
- **Delete confirmation is browser-side** — `window.confirm()` is not testable in API integration tests. Frontend E2E tests would need to handle this, but that's out of scope for the tester agent.
- **No service layer** means all logic is in the router — integration tests via TestClient cover the full stack. No unit tests needed for services.

## Open Questions ⚠️

1. **Should `client_name` have a max length constraint?**
   Default if not answered: varchar(255) — matches the pattern from architecture.md. Pydantic `max_length=255` validation on FeedbackCreate and FeedbackUpdate.

2. **Should `summary` have a max length constraint?**
   Default if not answered: varchar(500) — long enough for a meaningful summary, short enough to display in a table column without truncation issues.

3. **Should the list endpoint return items sorted by `created_at DESC`?**
   Default if not answered: Yes — ui-spec says "Rows sorted by date logged descending (newest first)". The backend will apply `.order_by(FeedbackItem.created_at.desc())` so the frontend doesn't need to sort.

4. **Should `detail` field in FeedbackUpdate be distinguishable between "not sent" and "explicitly set to null"?**
   Default if not answered: No — `exclude_unset=True` handles this. If `detail` is not in the request body, it won't be updated. If it's explicitly sent as `null`, it will be set to null. This is standard Pydantic partial update behavior.
