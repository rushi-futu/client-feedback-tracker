## Shared Types

These exact types must be used by both agents. Both copy verbatim. No interpretation.

### Backend — Pydantic schemas (backend/app/schemas/feedback.py)

```python
from enum import Enum
from pydantic import BaseModel
from datetime import datetime


class Theme(str, Enum):
    ux = "UX"
    performance = "Performance"
    support = "Support"
    pricing = "Pricing"
    communication = "Communication"


class FeedbackStatus(str, Enum):
    open = "open"
    in_progress = "in_progress"
    actioned = "actioned"


class FeedbackCreate(BaseModel):
    client_name: str
    summary: str
    detail: str | None = None
    theme: Theme
    status: FeedbackStatus


class FeedbackUpdate(BaseModel):
    client_name: str | None = None
    summary: str | None = None
    detail: str | None = None
    theme: Theme | None = None
    status: FeedbackStatus | None = None


class FeedbackRead(BaseModel):
    id: int
    client_name: str
    summary: str
    detail: str | None
    theme: Theme
    status: FeedbackStatus
    created_at: datetime
    updated_at: datetime

    model_config = {"from_attributes": True}
```

**Backend service logic**:

No dedicated service layer for this feature — business logic is minimal (enum validation handled by Pydantic, sorting handled in query). All logic lives in the router, following the existing pattern from api-patterns.md.

- `list_feedback(db)`:
  - Query `Feedback` table, order by `created_at` descending
  - Return all rows
  - Response: `list[FeedbackRead]`

- `get_feedback(feedback_id, db)`:
  - Query `Feedback` by `id`
  - If not found: raise HTTPException 404, detail "Feedback not found"
  - Return row
  - Response: `FeedbackRead`

- `create_feedback(body: FeedbackCreate, db)`:
  - Create `Feedback(**body.model_dump())`
  - `db.add()`, `db.commit()`, `db.refresh()`
  - Return created row
  - Response: `FeedbackRead`, status 201

- `update_feedback(feedback_id, body: FeedbackUpdate, db)`:
  - Query `Feedback` by `id`
  - If not found: raise HTTPException 404, detail "Feedback not found"
  - Loop `body.model_dump(exclude_unset=True)` → `setattr(feedback, key, value)`
  - `db.commit()`, `db.refresh()`
  - Return updated row
  - Response: `FeedbackRead`

- `delete_feedback(feedback_id, db)`:
  - Query `Feedback` by `id`
  - If not found: raise HTTPException 404, detail "Feedback not found"
  - `db.delete(feedback)`, `db.commit()`
  - Response: 204, no body

**Frontend component tree** (this slice):
```
feedback/
├── new/
│   └── page.tsx → NewFeedbackPage (server — no data fetch needed)
│       ├── "Log Feedback" heading
│       ├── Cancel link → /feedback
│       └── FeedbackForm (client — create mode, calls createFeedback)
│
└── [id]/
    └── page.tsx → FeedbackDetailPage (server — fetches getFeedback(id))
        ├── "Edit Feedback" heading
        ├── Cancel link → /feedback
        ├── FeedbackForm (client — edit mode, pre-populated, calls updateFeedback)
        └── Delete button (client — confirm dialog → calls deleteFeedback)
```

FeedbackForm component details:
- Props: `mode: "create" | "edit"`, `initialData?: FeedbackItem`, `onSuccess: () => void`
- Create mode: all fields empty except status defaults to "open". Submit calls `createFeedback`.
- Edit mode: all fields pre-populated from `initialData`. Submit calls `updateFeedback`.
- Validation: client_name and summary required — show inline error message beneath field if empty on submit.
- On success: both modes call `onSuccess` which triggers `router.push("/feedback")`.
- Loading state: submit button shows "Saving..." and is disabled during API call.
- Error state: API error displayed as banner above form.

Delete button details:
- Secondary/destructive style button below the form
- On click: `window.confirm("Are you sure you want to delete this feedback?")` — if confirmed, calls `deleteFeedback(id)`, then `router.push("/feedback")`

**Contract endpoints covered**: GET /feedback/{id}, POST /feedback/, PATCH /feedback/{id}, DELETE /feedback/{id}

## Decisions Made

1. **No service layer — router-only for all CRUD operations**
   Why: Business logic is minimal (no scoring, no computed fields, no cross-entity operations). Enum validation is handled by Pydantic. Sorting is a single `order_by` clause. Adding a service layer adds indirection with no benefit.
   Alternative rejected: Service layer per architecture.md convention — rejected because that pattern exists for complex logic (assignment_scorer, coverage_gap). CRUD with no business rules doesn't warrant it.

2. **All 5 backend endpoints built in Slice 1, not split across slices**
   Why: The endpoints are simple CRUD with identical patterns. Splitting them would create artificial boundaries. Having all endpoints ready means Slice 2 (frontend forms) can use any endpoint immediately.
   Alternative rejected: One slice per endpoint — too granular, creates unnecessary handoff overhead for trivial code.

3. **Client-side filtering with a client component wrapper**
   Why: The ui-spec explicitly states "Filters on `/feedback` are client-side — no round trip needed if all items loaded" and "No pagination — all items loaded on page mount". A server component fetches the data, then passes it to a client component that manages filter state.
   Alternative rejected: Server-side filtering with query params — contradicts the ui-spec.

4. **FeedbackListClient wrapper component between server page and FilterBar/FeedbackTable**
   Why: Next.js 15 server components cannot use useState. FilterBar and FeedbackTable both need shared filter state. A single client boundary component (`FeedbackListClient`) owns the filter state and passes filtered data to FeedbackTable and filter handlers to FilterBar.
   Alternative rejected: Making the page itself a client component — loses server-side data fetching benefits. Using separate client components with context — overengineered for local state.

5. **FeedbackForm as a single component with mode prop, not two separate forms**
   Why: Create and edit forms share identical fields, validation, and layout. Only the submit handler and pre-population differ. A `mode` prop cleanly separates these concerns.
   Alternative rejected: Separate CreateFeedbackForm and EditFeedbackForm — would duplicate 90% of the form code.

6. **`window.confirm()` for delete confirmation**
   Why: The ui-spec explicitly says "browser confirm dialog is acceptable". No need for a custom modal.
   Alternative rejected: Custom shadcn AlertDialog — would work but is overengineered given the explicit ui-spec guidance.

7. **Table name `feedback` (not `feedbacks`)**
   Why: "Feedback" is an uncountable noun in English. `feedback` reads more naturally as a table name than `feedbacks`.
   Alternative rejected: `feedbacks` per strict plural convention — grammatically awkward.

8. **Integer PK, not UUID**
   Why: The approved contract specifies `id: integer`. The ui-spec mentioned UUID but the contract overrides it — contract is the source of truth.
   Alternative rejected: UUID — would contradict the approved contract.

9. **Frontend components under `components/feedback/` not `components/forms/` or `components/board/`**
   Why: The existing architecture uses domain-based component grouping (`board/`, `forms/`). Since feedback components include both display (table) and form elements, a single `feedback/` domain folder is cleaner than splitting across `forms/` and a new display folder.
   Alternative rejected: Split across `forms/FeedbackForm.tsx` and `board/FeedbackTable.tsx` — creates artificial separation for a single-domain feature.

## Testability Notes

- **No service layer to test in isolation** — all logic is in router functions. Integration tests via TestClient are the appropriate test strategy. The tester agent should test through HTTP, not call router functions directly.
- **Enum validation is handled by Pydantic** — tester should verify that invalid enum values return 422, but doesn't need to unit test enum classes separately.
- **PATCH with `exclude_unset=True`** — tester should verify that sending `{"status": "actioned"}` doesn't null out other fields. This is the most likely source of bugs.
- **`order_by(created_at.desc())`** — tester should create multiple items and verify list order. Timing sensitivity: items created in rapid succession may share timestamps, so tester should either use explicit sleeps or check that order is at least non-ascending.
- **SQLite vs PostgreSQL** — tests use SQLite (per conftest pattern). SAEnum behavior differs slightly between SQLite and PostgreSQL. If enum validation tests fail unexpectedly, this is a known divergence to investigate.
- **Client-side filtering** — cannot be tested via backend integration tests. Frontend tests (if any) should cover FilterBar filter application and empty states.

## Open Questions

1. **Should `client_name` have a max length constraint?**
   Default if not answered: Yes, `varchar(255)` — same as summary. Pydantic schema does not add `max_length` validation (relies on DB constraint). If the PM wants stricter validation with a user-facing error, this can be added later.

2. **Should the list endpoint sort by `created_at` or `updated_at`?**
   Default if not answered: `created_at` descending — matches ui-spec "Date Logged" column header, which implies creation date not last-modified date.

3. **Should shadcn components be installed via CLI or manually created?**
   Default if not answered: Backend agent doesn't touch this. Frontend agent should use `npx shadcn@latest init` and `npx shadcn@latest add` to install needed components (button, input, select, table, card, badge). This ensures correct setup of shadcn's CSS variables and component configs.
