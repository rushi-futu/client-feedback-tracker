**Backend service logic**:

No separate service layer for this feature. The logic is pure CRUD with no business rules — routers interact directly with the DB session per the existing api-patterns. Adding a service layer would be over-engineering for straight pass-through CRUD.

Router pseudocode:

- `list_feedback(db)`:
  - Query all FeedbackItem rows, order by created_at DESC
  - Return list

- `get_feedback(feedback_id: UUID, db)`:
  - Query FeedbackItem by id
  - If not found → 404 "Feedback item not found"
  - Return item

- `create_feedback(body: FeedbackCreate, db)`:
  - Create FeedbackItem from body.model_dump()
  - Generate UUID for id (via Python uuid4 default on model)
  - Add, commit, refresh
  - Return item (201)

- `update_feedback(feedback_id: UUID, body: FeedbackUpdate, db)`:
  - Query FeedbackItem by id
  - If not found → 404
  - Loop body.model_dump(exclude_unset=True) → setattr each field
  - Commit, refresh
  - Return item

- `delete_feedback(feedback_id: UUID, db)`:
  - Query FeedbackItem by id
  - If not found → 404
  - Delete, commit
  - Return 204 (no body)

## Decisions Made

1. **UUID primary key using Python `uuid.uuid4` default on model, not database-generated**
   Why: Contract specifies `id: uuid`. Python-side generation is simpler, portable across databases (works with SQLite in tests), and avoids DB-specific UUID extensions.
   Alternative rejected: Database-level `gen_random_uuid()` — requires PostgreSQL extension, less portable for test fixtures using SQLite.

2. **No service layer — router handles DB operations directly**
   Why: This feature is pure CRUD with zero business logic. Every endpoint is a direct mapping from HTTP to database operation. A service layer would be an empty pass-through adding files and indirection with no benefit.
   Alternative rejected: Service layer per architecture.md convention — the convention says "services contain business logic". There is no business logic here. Adding a service file with functions that just call `db.query()` adds complexity without value. If business logic is added later (e.g., notifications on status change), a service layer should be introduced at that point.

3. **Client-side filtering only — no query parameters on GET /feedback/**
   Why: UI spec explicitly states "Filters on `/feedback` are client-side — no round trip needed if all items loaded" and "No pagination — all items loaded on page mount". The API returns all items; the frontend filters in-memory.
   Alternative rejected: Server-side filtering with query params — contradicts the ui-spec's explicit design decision.

4. **FeedbackListPage is a client component (not server component)**
   Why: The page needs `useState` for filter state and real-time filtering. While the data fetch could be done server-side, the filtering interaction requires client-side state. Using `useEffect` to fetch on mount keeps the component self-contained and avoids a hydration mismatch between server-rendered unfiltered content and client-filtered content.
   Alternative rejected: Server component page with client FilterBar child — would require passing all data down and lifting filter state, adding complexity for no benefit since there's no SEO requirement for this internal tool.

5. **FeedbackForm is a single reusable component with a `mode` prop**
   Why: Create and edit forms have identical fields per ui-spec. The only differences are: button text, pre-populated values, and which API function to call. A single component with `mode: "create" | "edit"` avoids duplication.
   Alternative rejected: Separate CreateForm and EditForm — violates DRY with identical field sets.

6. **Theme enum values use PascalCase (UX, Performance, etc.) not snake_case**
   Why: The ui-spec and contract both specify the values as `UX | Performance | Support | Pricing | Communication`. These are display-ready labels, not code identifiers. Keeping them PascalCase means no transformation needed for display.
   Alternative rejected: snake_case values with display mapping — adds unnecessary mapping logic.

7. **Delete confirmation uses `window.confirm()` browser dialog**
   Why: UI spec explicitly says "browser confirm dialog is acceptable". No need for a custom modal component.
   Alternative rejected: Custom Dialog component — over-engineering given the explicit spec guidance.

8. **Frontend component directory is `components/feedback/` not `components/board/` or `components/forms/`**
   Why: The architecture.md shows `board/` and `forms/` as examples from the story-assignment project. For this project, grouping by domain (`feedback/`) is cleaner since all components belong to a single feature. FilterBar, FeedbackTable, and FeedbackForm are all feedback-domain components.
   Alternative rejected: Splitting into `board/FeedbackTable` and `forms/FeedbackForm` — artificial separation for a single-domain app.

9. **SQLAlchemy model uses `SAEnum` with Python enum for theme and status columns**
   Why: Follows data-patterns.md convention. Keeps DB and Python types in sync. SAEnum creates a CHECK constraint in PostgreSQL ensuring only valid values are stored.
   Alternative rejected: Plain VARCHAR with application-level validation only — risks invalid data in the database.

10. **shadcn/ui components installed as needed (Input, Select, Button, Table, Textarea, Card)**
    Why: Frontend-patterns.md mandates shadcn for all primitives. The frontend agent should run `npx shadcn@latest init` and then add needed components.
    Alternative rejected: Custom-built form controls — violates the shadcn-first convention.

## Testability Notes

- **All endpoints are directly testable via HTTP** using TestClient + SQLite override (per data-patterns.md test database pattern). No mocking needed.
- **UUID generation** — tests cannot predict IDs. Must capture `id` from POST response to use in subsequent GET/PATCH/DELETE calls.
- **Client-side filtering** — not testable via API. Requires frontend component-level tests (Vitest + testing-library). The tester agent should test that FilterBar correctly filters a known dataset.
- **PATCH partial update** — important to test that sending only `{status: "actioned"}` does not null out other fields. This is a common CRUD bug.
- **Enum validation** — Pydantic handles this automatically (returns 422 for invalid enum values). Worth testing that invalid theme/status values are rejected.
- **Delete cascade** — no foreign keys reference feedback_items, so delete is clean. No cascade concerns.
- **SQLite vs PostgreSQL** — SAEnum with Python enums works on both, but SQLite doesn't enforce CHECK constraints. Tests should validate enum values at the application level, not rely on DB constraints.

## Open Questions ⚠️

1. **Should the list endpoint support sorting by columns other than created_at?**
   Default if not answered: No. The API returns items sorted by `created_at DESC` only. The ui-spec says "Rows sorted by date logged descending (newest first)" with no mention of user-sortable columns. Client-side sorting could be added to the frontend later without API changes since all items are loaded.

2. **Should string fields (client_name, summary) have maximum length validation?**
   Default if not answered: Yes. `client_name` max 255 chars, `summary` max 500 chars, matching the VARCHAR constraints in the data model. Pydantic schemas will enforce via `Field(max_length=...)`. The ui-spec does not specify limits, but unbounded strings are a data quality risk.

3. **Should the frontend use optimistic updates or wait for server confirmation?**
   Default if not answered: Wait for server confirmation. Redirect to `/feedback` only after the API responds with success. This is simpler, avoids rollback complexity, and is appropriate for an internal tool where latency is not critical.
