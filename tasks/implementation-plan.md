#### Backend service logic (`backend/app/services/feedback.py`)

```
Imports: Session from sqlalchemy.orm, uuid, datetime, FeedbackItem model,
         FeedbackItemCreate schema, FeedbackItemUpdate schema

FUNCTION list_feedback(db: Session) -> list[FeedbackItem]:
  RETURN db.query(FeedbackItem)
           .order_by(FeedbackItem.created_at.desc())
           .all()

FUNCTION get_feedback(db: Session, item_id: uuid.UUID) -> FeedbackItem | None:
  RETURN db.query(FeedbackItem)
           .filter(FeedbackItem.id == str(item_id))
           .first()

FUNCTION create_feedback(db: Session, body: FeedbackItemCreate) -> FeedbackItem:
  item = FeedbackItem(
    id=str(uuid.uuid4()),
    **body.model_dump()
  )
  db.add(item)
  db.commit()
  db.refresh(item)
  RETURN item

FUNCTION update_feedback(db: Session, item_id: uuid.UUID, body: FeedbackItemUpdate)
    -> FeedbackItem | None:
  item = get_feedback(db, item_id)
  IF item is None:
    RETURN None
  FOR key, value IN body.model_dump(exclude_unset=True).items():
    setattr(item, key, value)
  item.updated_at = datetime.utcnow()   ← set explicitly for cross-dialect safety
  db.commit()
  db.refresh(item)
  RETURN item

FUNCTION delete_feedback(db: Session, item_id: uuid.UUID) -> bool:
  item = get_feedback(db, item_id)
  IF item is None:
    RETURN False
  db.delete(item)
  db.commit()
  RETURN True
```

#### Backend router logic (`backend/app/routers/feedback.py`)

```
router = APIRouter(prefix="/feedback", tags=["feedback"])

GET /
  CALL list_feedback(db)
  RETURN list[FeedbackItemRead], status 200

POST /
  CALL create_feedback(db, body)
  RETURN FeedbackItemRead, status 201

GET /{item_id: uuid.UUID}
  CALL get_feedback(db, item_id)
  IF None: raise HTTPException(404, "Feedback item not found")
  RETURN FeedbackItemRead, status 200

PATCH /{item_id: uuid.UUID}
  CALL update_feedback(db, item_id, body)
  IF None: raise HTTPException(404, "Feedback item not found")
  RETURN FeedbackItemRead, status 200

DELETE /{item_id: uuid.UUID}
  CALL delete_feedback(db, item_id)
  IF False: raise HTTPException(404, "Feedback item not found")
  RETURN None, status 204
```

Note: FastAPI validates `{item_id}` as `uuid.UUID` automatically; a malformed UUID
returns 422 without any extra code.

## Decisions Made

1. **UUID generated in Python, stored as String(36)**
   Why: `uuid.UUID` is specified in the contract for `id`. SQLite (used in tests per data-patterns.md)
   does not have a native UUID column type. Storing as `String(36)` (or `Uuid(as_uuid=False)` in
   SQLAlchemy 2.0) ensures the same model works in both PostgreSQL (production) and SQLite (tests).
   Pydantic's `FeedbackItemRead` declares `id: UUID` and coerces the string to a UUID object on read.
   UUID value is generated in Python with `str(uuid.uuid4())` at insert time.
   Alternative rejected: `sqlalchemy.dialects.postgresql.UUID(as_uuid=True)` — breaks SQLite test
   DB; requires a TypeDecorator workaround that adds maintenance burden.

2. **Thin service layer for simple CRUD**
   Why: Architecture.md rule is unambiguous — "never put SQL in a router." The service layer
   (`services/feedback.py`) holds all five DB operations. The router holds only HTTP logic (status
   codes, 404 raises, response models). This boundary makes each layer independently testable.
   Alternative rejected: Inline DB operations in the router — violates the module boundary rule and
   makes HTTP logic harder to read.

3. **`updated_at` set explicitly in service on PATCH**
   Why: SQLAlchemy's `onupdate=func.now()` relies on the DB server-side function. In SQLite (test DB),
   this may not fire on a Python-level `setattr` + `commit`. To guarantee consistent behaviour across
   dialects, the `update_feedback` service function explicitly sets `item.updated_at = datetime.utcnow()`
   before commit.
   Alternative rejected: Rely solely on `onupdate` — works in PostgreSQL but is unreliable in SQLite
   tests, causing T-04 to produce false passes.

4. **Enums defined in `schemas/feedback.py`, imported by `models/feedback.py`**
   Why: Follows the project convention established in api-patterns.md — the Brief model imports
   `Priority` and `BriefStatus` from `app.schemas.brief`. Single source of truth for enum values.
   No risk of the DB constraint and the API schema diverging.
   Alternative rejected: Defining enums in `models/feedback.py` and importing into schemas —
   inverts the correct dependency direction (schemas should not depend on models) and risks circular
   imports.

5. **`FeedbackListClient` wrapper owns filter state**
   Why: Next.js 15 server components cannot hold `useState`. `FeedbackListPage` must remain a server
   component so it fetches data without a loading spinner on initial load. A thin `FeedbackListClient`
   wrapper receives `allItems` as a prop from the server component and owns filter state internally.
   This is the standard Next.js 15 server/client split pattern.
   Alternative rejected: Making the entire `FeedbackListPage` a client component — requires a
   `useEffect` fetch on mount, loses server-side data loading, introduces unnecessary loading state.

6. **Separate `FeedbackDetailClient` / `NewFeedbackClient` wrappers (not server pages calling hooks)**
   Why: Same reasoning as Decision 5. The pages fetch data server-side and pass it to thin client
   wrappers that own async mutations and `router.push()`. `window.confirm()` (for delete) is a browser
   API that requires a client component.
   Alternative rejected: Fully client pages — unnecessary; these pages have no real-time update needs.

7. **`FeedbackForm` is a single shared component with `mode` prop**
   Why: Create and edit forms are identical except for submit label and initial values. A single
   component eliminates duplication. The `mode` prop gates validation (required-field check only
   in create mode, not on edit) and submit label.
   Alternative rejected: Two separate `CreateFeedbackForm` and `EditFeedbackForm` components —
   identical field logic duplicated in two places; one bug fix would require two edits.

8. **`DeleteButton` as a separate UI element in `FeedbackDetailClient`, not inside `FeedbackForm`**
   Why: The delete action is destructive and architecturally separate from the save action. Keeping it
   outside `FeedbackForm` means the form component has a single responsibility (editing fields and
   submitting). The delete handler with `window.confirm()` lives in the page-level client wrapper.
   Alternative rejected: Delete button inside `FeedbackForm` — would require passing a delete handler
   as a prop to a form component, muddying its create/edit responsibility.

9. **Component directory: `frontend/src/components/feedback/`**
   Why: The codebase architecture uses domain directories under `components/` (e.g. `board/`, `forms/`
   in the story-assignment reference project). A `feedback/` directory groups all feedback-specific
   components. The existing `board/` and `forms/` directories are story-assignment domain names, not
   generic categories for this project.
   Alternative rejected: `components/forms/` for FeedbackForm — mixes the form/non-form split
   unnecessarily for a single-resource feature.

10. **Client-side filtering with no query params**
    Why: Explicitly specified in `design/ui-spec.md` § Interactions: "Filters on `/feedback` are
    client-side — no round trip needed if all items loaded." The contract GET /feedback/ has no query
    parameters. No pagination is required.
    Alternative rejected: Server-side filtering via query params — contradicts both the ui-spec and
    the approved contract.

## Testability Notes

- **No external service dependencies** — the feedback service is pure CRUD with no external calls
  (no scoring services, no notifications, no integrations). All five endpoints are straightforwardly
  testable with the SQLite test DB pattern from data-patterns.md (`conftest.py` override pattern).

- **UUID path params** — test fixtures must generate valid UUID strings: `str(uuid.uuid4())`.
  The test client sends them as path segments; FastAPI parses them as `uuid.UUID` automatically.
  Testing a malformed UUID path (e.g. `/feedback/not-a-uuid`) should return 422 without extra code.

- **PATCH `exclude_unset` semantics** — the most failure-prone line in the service is
  `body.model_dump(exclude_unset=True)`. If this is accidentally written as `model_dump()` (without
  the flag), every PATCH will overwrite unspecified fields with null. The tester must verify that a
  PATCH sending only `{"status": "actioned"}` does not clear `client_name` or `summary`.

- **`detail: null` clears the field** — per contract notes, sending `"detail": null` in a PATCH body
  must set the DB column to null. This works correctly only if `detail` is included in the parsed body
  (i.e. the client explicitly sends `"detail": null`, not omits the key). The tester should fire this
  case specifically.

- **`updated_at` on PATCH** — because `updated_at` is set explicitly in the service (Decision 3),
  tests can reliably assert that `updated_at` changes after a PATCH. If the tester sees `updated_at`
  unchanged after a PATCH, it means the service forgot the explicit assignment.

- **Empty list vs 404** — GET /feedback/ on an empty DB must return `[]` with status 200, not 404.
  This is an easy test but historically confused by developers expecting a "no content" response.
  Flag for T-01.

- **Synchronous endpoints, no pytest-asyncio needed** — standard FastAPI + SQLAlchemy session
  pattern; `TestClient` from `httpx` works without any async test setup.

## Open Questions ⚠️

1. **`updated_at` dialect behaviour**
   The implementation plan mandates explicit `item.updated_at = datetime.utcnow()` in the
   `update_feedback` service function (Decision 3). This resolves the SQLite `onupdate` gap.
   However, in PostgreSQL production the value will be set in Python (UTC) rather than by the DB
   server clock. These will be within milliseconds of each other. If the team requires DB-authoritative
   timestamps in production, the backend agent should add a `RETURNING updated_at` approach.
   **Default if not answered**: set `updated_at` explicitly in Python as specified above — acceptable
   for this application's requirements.

2. **PostgreSQL connection string for alembic.ini and config.py**
   The default DATABASE_URL follows the project pattern from api-patterns.md.
   **Default if not answered**: `postgresql://postgres:postgres@localhost:5432/feedback_tracker`
   (database name `feedback_tracker` to match this project, not `storyboard`).

3. **shadcn/ui initialisation**
   The frontend patterns reference shadcn/ui primitives (Button, Select, etc.). This is a greenfield
   project — shadcn may not be installed.
   **Default if not answered**: The frontend agent runs `npx shadcn@latest init` with default
   settings (TypeScript, Tailwind, src/ directory, app router) as its first step before writing any
   component, then adds required components (`npx shadcn@latest add button select input textarea`).

4. **`router.push` after delete — timing**
   After `deleteFeedback(id)` resolves, the client calls `router.push("/feedback")`. The list page
   will refetch on navigation. Since the list page is a server component, the deleted item will not
   appear after redirect. No cache invalidation is needed beyond the navigation itself.
   **Default if not answered**: rely on server component re-fetch on navigation (no additional cache
   busting); acceptable for this use case.
