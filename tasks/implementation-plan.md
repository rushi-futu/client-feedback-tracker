### Backend — Pydantic schemas (`app/backend/app/schemas/feedback.py`)

```python
from pydantic import BaseModel
from datetime import datetime
from enum import Enum


class Theme(str, Enum):
    UX = "UX"
    Performance = "Performance"
    Support = "Support"
    Pricing = "Pricing"
    Communication = "Communication"


class Status(str, Enum):
    Open = "Open"
    In_Progress = "In Progress"
    Actioned = "Actioned"


class FeedbackCreate(BaseModel):
    client_name: str
    summary: str
    detail: str | None = None
    theme: Theme
    status: Status = Status.Open


class FeedbackUpdate(BaseModel):
    client_name: str | None = None
    summary: str | None = None
    detail: str | None = None
    theme: Theme | None = None
    status: Status | None = None


class FeedbackRead(BaseModel):
    id: int
    client_name: str
    summary: str
    detail: str | None
    theme: Theme
    status: Status
    date_logged: datetime
    updated_at: datetime

    model_config = {"from_attributes": True}
```

**Backend service logic**:

No dedicated service layer for v1. All CRUD logic lives directly in the router, following the existing codebase pattern (see api-patterns.md). Business logic is simple enough that extraction to a service would be over-engineering.

Router functions — pseudocode:

1. **`list_feedback(search?, theme?, status?, db)`** → `list[FeedbackRead]`
   - Start with `db.query(Feedback)`
   - If `search` is provided: add `.filter(Feedback.client_name.ilike(f"%{search}%"))`
   - If `theme` is provided: add `.filter(Feedback.theme == theme)`
   - If `status` is provided: add `.filter(Feedback.status == status)`
   - Order by `Feedback.date_logged.desc()` (newest first, matching ui-spec table)
   - Return `.all()`

2. **`get_feedback(feedback_id, db)`** → `FeedbackRead | 404`
   - Query by id
   - If not found: raise HTTPException 404 with detail "Feedback item not found"
   - Return feedback

3. **`create_feedback(body: FeedbackCreate, db)`** → `FeedbackRead` (201)
   - Create Feedback instance from `body.model_dump()`
   - `db.add()`, `db.commit()`, `db.refresh()`
   - Return created feedback

4. **`update_feedback(feedback_id, body: FeedbackUpdate, db)`** → `FeedbackRead | 404`
   - Query by id, 404 if not found
   - Iterate `body.model_dump(exclude_unset=True)` and `setattr()` each field
   - `db.commit()`, `db.refresh()`
   - Return updated feedback

5. **`delete_feedback(feedback_id, db)`** → `204 | 404`
   - Query by id, 404 if not found
   - `db.delete()`, `db.commit()`
   - Return no content

**Data fetching strategy**: The list page uses a **client-side fetch pattern**. The `FeedbackListClient` component fetches `/feedback/` on mount and re-fetches when filters change. This is necessary because filters are interactive (search typing, dropdown selection) and the FilterBar state drives re-fetches. No server component data fetching on the list page — the client component owns the full lifecycle.

**Contract endpoints covered**: All 5 endpoints are implemented in the backend. The list page UI consumes `GET /feedback/` with filter params only.

**Form validation logic** (pseudocode — frontend only):
- Track `touched` state per field (set on blur)
- Track `errors` map: `{ fieldName: errorMessage }`
- On blur of required field: if empty, set error "This field is required"
- On submit: validate all required fields. If any errors, set all as touched, do not submit.
- Show red border (`border-red-500`) + error message (`text-red-500 text-sm`) below field when `touched && error`
- Clear field error when user types (onChange)

## Decisions Made

1. **No service layer for v1**
   Why: All operations are straightforward CRUD with no business logic beyond filtering. The existing codebase pattern (api-patterns.md) puts simple DB queries directly in routers. Extracting to a service would add indirection with no benefit.
   Alternative rejected: Service layer with repository pattern — over-engineered for simple CRUD.

2. **Client-side fetching for the list page (not server component fetch)**
   Why: The FilterBar requires interactive state (search input, dropdown selections) that drives re-fetches. A server component cannot manage this client-side interactivity. The list page wraps everything in a `FeedbackListClient` "use client" component that fetches on mount and re-fetches when filters change.
   Alternative rejected: Server component with URL-based search params — would work but adds complexity with router.push/replace for every filter change, and the latency of full page transitions is worse UX for real-time search filtering.

3. **Separate ThemeBadge and StatusBadge components**
   Why: Each has its own colour mapping (5 theme colours, 3 status colours) defined in ui-spec. Keeping them separate is cleaner than a generic badge with conditional colour logic.
   Alternative rejected: Single GenericBadge with a colour map prop — workable but less readable.

4. **Single FeedbackForm component for both create and edit modes**
   Why: The ui-spec explicitly describes the form as the same component in both modes, differing only in pre-population, button text, and the presence of the Delete button. A `mode` prop controls this.
   Alternative rejected: Separate CreateForm and EditForm — would duplicate field definitions and validation logic.

5. **Enum values use display strings (e.g. "In Progress") not snake_case**
   Why: The contract defines Status values as `"Open" | "In Progress" | "Actioned"` — these are user-facing display values. Both Python enum and TypeScript type use these exact strings. The backend SAEnum stores these strings in the DB.
   Alternative rejected: snake_case enum values with display mapping — adds a translation layer on both sides for no benefit, and contradicts the approved contract.

6. **Order feedback by date_logged descending (newest first)**
   Why: The ui-spec prototype shows the most recent items at the top. This is the natural expectation for a feedback log.
   Alternative rejected: No explicit ordering — would result in insertion order which is similar but not guaranteed after updates.

7. **Feedback table columns use `date_logged` not `updated_at` for the "Date Logged" column**
   Why: The ui-spec explicitly names the column "Date Logged" and says it is "auto-set on create". The `updated_at` field exists in the schema but is not displayed in the table per the ui-spec.
   Alternative rejected: Showing both dates — not in ui-spec, adding UI that wasn't designed.

8. **Greenfield backend scaffolding in Slice 1**
   Why: The app directory has only Dockerfiles. Slice 1 must create all foundational backend files: `database.py`, `config.py`, `main.py`, `__init__.py` files, `pyproject.toml`, alembic config, and the migration. These follow the patterns in architecture.md and api-patterns.md exactly.
   Alternative rejected: N/A — this is required, not a choice.

9. **Frontend scaffolding: shadcn components not included**
   Why: The ui-spec uses standard HTML elements (input, select, button, table). The components are simple enough that shadcn primitives (Button, Select, Input) could be used but are not required. The frontend agent should use shadcn/ui components where they exist in `components/ui/` but can fall back to styled HTML elements with Tailwind if shadcn components haven't been installed. The frontend agent should run `npx shadcn@latest init` and add needed components if the project doesn't already have them set up.
   Alternative rejected: Mandating specific shadcn components — the frontend agent knows the patterns and should decide which primitives to use.

10. **Catch-all 404 uses Next.js `not-found.tsx`**
    Why: Next.js 15 app router has built-in support for `not-found.tsx` at any level. A root-level `not-found.tsx` that redirects to `/feedback` handles the ui-spec requirement that unknown routes redirect to `/feedback`.
    Alternative rejected: Middleware-based redirect — heavier and less idiomatic for Next.js app router.

## Testability Notes

- **List filtering logic**: The filtering (search + theme + status) is inline in the router function. If the tester finds it hard to test filter combinations in isolation, the filtering logic can be extracted to a helper function. For now, integration tests via `TestClient` with query params are sufficient.
- **Enum validation**: Pydantic handles enum validation automatically. The tester should verify that invalid theme/status values return 422, not 500.
- **PATCH partial update**: The `exclude_unset=True` pattern means sending `{}` should be a valid no-op update. The tester should verify this edge case.
- **Delete idempotency**: Deleting an already-deleted item should return 404 (not 204). The tester should verify this.
- **SQLite in tests**: The test database pattern uses SQLite (see data-patterns.md conftest). SAEnum with string enums works with SQLite, but the tester should be aware that SAEnum behaviour may differ slightly between SQLite and PostgreSQL. If enum validation issues arise in tests, the conftest may need adjustment.
