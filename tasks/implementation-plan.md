from datetime import datetime
from enum import Enum


class Theme(str, Enum):
    UX = "UX"

    Support = "Support"
    Pricing = "Pricing"
    Communication = "Communication"

    Open = "Open"
    InProgress = "In Progress"
    Actioned = "Actioned"


class FeedbackCreate(BaseModel):
    client_name: str
    summary: str
    detail: str | None = None
    theme: Theme
    status: Status = Status.Open


class FeedbackUpdate(BaseModel):
    client_name: str | None = None
    summary: str | None = None
    detail: str | None = None
    theme: Theme | None = None
    status: Status | None = None


class FeedbackRead(BaseModel):
    id: int
    client_name: str
    summary: str
    detail: str | None = None
    theme: Theme
    status: Status
    date_logged: datetime

    model_config = {"from_attributes": True}

**Backend service logic**:

No service layer for MVP — router handles CRUD directly (per testability notes in current-feature.md). Logic for each endpoint:

1. `list_feedback(db) -> list[FeedbackRead]`
   - Query all rows from feedback_items
   - Order by date_logged descending (newest first)
   - Return list

2. `get_feedback(feedback_id: int, db) -> FeedbackRead`
   - Query feedback_items where id == feedback_id
   - If not found: raise HTTPException(404, "Feedback item not found")
   - Return item

3. `create_feedback(body: FeedbackCreate, db) -> FeedbackRead`
   - Create Feedback model from body.model_dump()
   - Add to session, commit, refresh
   - Return created item (status 201)
   - Pydantic handles 422 automatically for invalid/missing fields

4. `update_feedback(feedback_id: int, body: FeedbackUpdate, db) -> FeedbackRead`
   - Query feedback_items where id == feedback_id
   - If not found: raise HTTPException(404, "Feedback item not found")
   - Iterate body.model_dump(exclude_unset=True) — only update provided fields
   - Commit, refresh
   - Return updated item

5. `delete_feedback(feedback_id: int, db) -> None`
   - Query feedback_items where id == feedback_id
   - If not found: raise HTTPException(404, "Feedback item not found")
   - Delete from session, commit
   - Return 204 No Content

## Decisions Made

1. **No service layer for MVP**
   Why: All endpoints are straightforward CRUD with no business logic. Adding a service layer adds indirection with no benefit. The current-feature.md testability notes explicitly endorse this.
   Alternative rejected: Service layer per-endpoint — would be premature abstraction. If rules like "Actioned items become read-only" are added later, extract to a service then.

2. **Router handles DB queries directly**
   Why: Follows the existing codebase pattern in api-patterns.md exactly (briefs router does the same). Consistency > purity for MVP.
   Alternative rejected: Repository pattern — overkill for single-table CRUD.

3. **Client-side filtering only (no query params on GET /feedback/)**
   Why: The ui-spec explicitly states "Filtering is client-side and immediate (no submit button)". The contract reflects this — GET /feedback/ returns all items, frontend filters.
   Alternative rejected: Server-side filtering with query params — would contradict the ui-spec's design intent and add unnecessary complexity for MVP scale.

4. **FeedbackForm as a single component with mode prop (create/edit)**
   Why: Both forms have identical fields and validation. Only differences: pre-populated data, button text, delete button presence. A single component with a `mode` prop avoids duplication.
   Alternative rejected: Separate CreateFeedbackForm / EditFeedbackForm — 90% identical code, maintenance burden.

5. **Feedback list page as server component that passes data to client children**
   Why: Initial data fetch happens server-side (faster first paint). FilterBar and FeedbackTable are client components because they need useState for interactive filtering.
   Alternative rejected: Full client-side page with useEffect fetch — slower first paint, loading spinner flash.

6. **SQLAlchemy SAEnum for theme and status columns**
   Why: Follows data-patterns.md convention. Keeps DB and Python enums in sync. PostgreSQL will enforce valid values at DB level.
   Alternative rejected: Plain String column with app-level validation only — allows invalid data in DB.

7. **`date_logged` instead of `created_at` + `updated_at`**
   Why: The contract specifies `date_logged` only. The ui-spec shows no "updated at" anywhere. Adding `updated_at` would deviate from the contract for no UI benefit.
   Alternative rejected: Adding both `created_at` and `updated_at` per data-patterns.md convention — would add a field not in the contract and not displayed in the UI.

8. **Component directory: `components/feedback/` not `components/board/` or `components/forms/`**
   Why: The frontend-patterns.md shows `board/` and `forms/` for the story assignment app. This is a different domain. Using `feedback/` as the domain directory keeps all feedback components together, which is cleaner for a single-domain app.
   Alternative rejected: Splitting into `board/` (table) and `forms/` (form) — artificial separation for only 5 components.

9. **Frontend needs scaffolding files created**
   Why: The frontend directory currently has no package.json, tsconfig.json, next.config, or tailwind config. These foundational files must be created by the frontend agent before any components. The Dockerfile exists but needs source files to build against.
   Alternative rejected: Assuming these files exist — they don't. The agent must create them.

10. **Backend needs all foundational files created**
    Why: The backend directory currently has empty `models/`, `schemas/`, `routers/` dirs with only `__pycache__` remnants. `main.py`, `database.py`, `config.py`, `pyproject.toml`, `alembic.ini` must all be created.
    Alternative rejected: Assuming these files exist — they don't.

## Testability Notes

- **No service layer to test in isolation**: All business logic is in the router. Integration tests via TestClient are the appropriate test strategy. If business rules are added later, extract to a service.
- **`date_logged` uses server_default**: Tests must not assert exact datetime values. Assert `date_logged` is a valid ISO datetime, or assert ordering (item1.date_logged >= item2.date_logged).
- **SAEnum with spaces**: `Status.InProgress` has value `"In Progress"` (with space). Tests must use the exact string `"In Progress"` in request/response bodies. This is a common source of test failures.
- **FeedbackUpdate partial updates**: PATCH with `exclude_unset=True` means sending `{"status": "Actioned"}` should only update status, leaving all other fields unchanged. Tests must verify non-provided fields are untouched.
- **Client-side filtering**: No backend test needed. Frontend tests should cover filter logic (search by client name, dropdown by theme/status, clear filters).
- **SQLite for tests vs PostgreSQL for prod**: SAEnum behaves differently on SQLite (stores as string, no actual ENUM type). Tests should still pass because SQLAlchemy handles the abstraction, but be aware of this if edge cases arise around enum validation.
